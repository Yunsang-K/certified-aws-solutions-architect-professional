# S3

## 스토리지 클래스(Storage Classes)

* **S3 Standard (기본)**

  * 객체(Object)는 **최소 3개 AZ**에 저장됨
  * **가용성(Availability) 11 9s(99.999999999%)** 제공(서비스 설명에서 흔히 이렇게 표현)
  * 복제는 **MD5 체크 + CRC**로 객체 이상을 감지
  * API로 객체를 Put 하면 **HTTP 200 OK** 응답
  * 과금:

    * S3에 저장된 **GB/월**
    * **데이터 전송(Outbound) GB당 요금** (Inbound는 무료)
    * **요청(Request) 1,000건당 요금**
    * **조회(리트리벌) 요금 없음**, **최소 보관 기간 없음**, **최소 객체 크기 없음**
  * 즉시 접근 가능(밀리초 지연), **정적 웹사이트 호스팅**에도 사용 가능
  * **자주 접근하는 데이터**에 적합

* **S3 Standard-IA (Infrequent Access)**

  * S3 Standard와 대부분 특성 동일: **3 AZ 복제**, 내구성 동일, (설계상) 가용성/지연 특성 유사, 퍼블릭 공개 가능
  * 과금:

    * 저장 단가는 더 저렴
    * 데이터 전송 요금은 Standard와 동일
    * **조회(리트리벌) 요금**: GB당 부과 → 자주 읽으면 총비용이 증가 가능
    * **최소 보관 30일**, **최소 과금 객체 크기 128KB**(128KB 미만도 128KB로 과금)
  * **장기 보관 + 드문 접근** 데이터에 적합

* **S3 One Zone-IA**

  * Standard-IA보다도 더 저렴(대신 가용성/HA 측면에서 불리)
  * 데이터는 **단일 AZ에만 저장**(AZ 간 복제 없음)
  * 과금:

    * Standard-IA와 유사: **30일 최소 보관**, **128KB 최소 과금**, **GB당 조회 요금**
    * **AZ가 정상인 한** 내구성은 동일 수준(단, AZ 장애 시 리스크 큼)
    * AZ 내부에서만 복제
  * **비핵심 데이터** 또는 **쉽게 재생성 가능한 데이터**에 적합(고가용성 목적에는 부적합)

* **S3 Glacier Instant Retrieval**

  * Standard-IA처럼 “즉시(밀리초) 접근”이지만

    * 저장은 더 저렴
    * 조회는 더 비쌈
    * 최소 보관 기간이 더 김
  * **분기 1회 수준으로 드물게 접근하지만 즉시 복원이 필요한 데이터**에 권장
  * **최소 보관 90일**

* **S3 Glacier Flexible Retrieval (구 S3 Glacier)**

  * Standard/Standard-IA와 동일한 복제 방식(멀티 AZ), 동일한 내구성 특성
  * 저장 비용은 Standard 대비 대략 **1/6 수준**(설명 관행)
  * Glacier 객체는 “콜드 데이터”로 간주(자주 접근하면 비효율)
  * Glacier 클래스의 객체는 **바로 퍼블릭 공개 불가**(복원/조회 절차 필요)
  * 조회 시 **Restore Job(복원 작업)**이 필요하며, 복원/조회는 과금 대상
  * 복원 시 객체가 **일시적으로 Standard-IA 영역에 올라왔다가** 일정 시간 후 제거될 수 있음(영구 복원 옵션도 가능)
  * 복원 작업 유형:

    * **Expedited**: 1~5분 (가장 비쌈)
    * **Standard**: 3~5시간
    * **Bulk**: 5~12시간 (저렴)
  * **최소 과금 40KB**, **최소 보관 90일**
  * **아카이브(연 1회 수준 접근)**에 적합

* **S3 Glacier Deep Archive**

  * “동결(frozen)”에 가까운 초저빈도 아카이브
  * **최소 과금 40KB**, **최소 보관 180일**
  * 퍼블릭 공개 불가, 접근은 Glacier와 유사(복원 필요)
  * 복원 작업 시간:

    * **Standard**: 최대 12시간
    * **Bulk**: 최대 48시간
  * **거의 접근하지 않는 장기 보관(컴플라이언스/백업 아카이브 등)**에 적합

* **S3 Intelligent-Tiering**

  * 내부적으로 **5개 티어**로 자동 계층화
  * 자주 접근: Frequent Access
  * 덜 접근: Infrequent Access
  * 매우 드묾: Archive 또는 Deep Archive(옵션)
  * 객체 이동을 사용자가 직접 관리할 필요가 없고 **자동으로 티어 이동**
  * 아카이브 티어(Archive/Deep Archive)는 **활성/비활성 선택 가능**
  * Frequent ↔ Infrequent 간 이동에는 조회 요금 없음
  * 대신 **모니터링/자동화 비용(객체 1,000개당 요금)** 발생
  * **접근 패턴이 불확실/가변적인 데이터**에 권장

## S3 라이프사이클 설정(S3 Lifecycle Configuration)

* 버킷에 **Lifecycle Rule**을 만들어 객체를 스토리지 클래스 간 이동(Transition)하거나 만료(Expiration)로 삭제 가능

* 라이프사이클 구성은 “버킷 전체” 또는 “특정 Prefix/Tag 집합”에 규칙을 적용하는 형태

* 규칙의 액션:

  * **Transition**: 일정 시간이 지나면 다른 클래스로 이동
  * **Expiration**: 객체/버전 삭제

* “접근 빈도” 기반 이동은 라이프사이클로는 불가(그건 Intelligent-Tiering이 담당)

* 라이프사이클은 “시간 경과” 기준으로 이동/삭제

* 고려 사항:

  * **One Zone-IA → Glacier Flexible/Deep Archive는 가능**, **Glacier Instant Retrieval로는 전환 불가**
  * 작은 객체는 Standard-IA/One Zone-IA 등에서 **최소 과금 크기 때문에 비효율**이 될 수 있음
  * 객체는 **Standard에 최소 30일** 있어야 IA 계열로 “규칙 기반 전환” 가능(단, 처음부터 IA로 업로드하는 건 가능)
  * 하나의 규칙으로 Standard-IA → Glacier까지 “바로” 연쇄 전환 불가:

    * IA 계열에서 **최소 30일** 체류 조건 때문에, 필요하면 **규칙을 2개로 분리**해야 함

## S3 복제(S3 Replication)

* 유형 2가지:

  * **CRR (Cross-Region Replication)**: 리전 간
  * **SRR (Same-Region Replication)**: 같은 리전 내

* 둘 다 **동일 계정/교차 계정** 복제 지원

* 교차 계정 복제 시 대상 계정에 **목적지 버킷 정책**으로 소스 계정 복제 권한 부여 필요

* 전체 복제 또는 Prefix/Tag 기반 일부 복제 가능

* 대상 버킷 저장 클래스 지정 가능(기본은 동일 클래스)

* 목적지 객체 소유권도 설정 가능(기본은 소스와 동일)

* **RTC(Replication Time Control)**: 활성화 시 **15분 내 복제 보장**(SLA 성격)

* 복제 고려 사항:

  * 기본적으로 **소급 복제(기존 객체) 안 됨**: 설정 이후 새 객체만 복제
  * **소스/대상 모두 버전닝 활성화 필요**
  * 기존 객체 복제는 **Batch Replication**을 별도로 구성해야 함
  * 기본은 **단방향**(소스 → 대상). 양방향은 별도 구성 필요
  * SSE-S3, SSE-KMS는 추가 설정 포함해 복제 가능. SSE-C도 지원(과거엔 제약이 있었던 케이스가 존재)
  * 소스 버킷 소유자는 복제 대상 객체에 대한 권한이 필요
  * **시스템 이벤트는 복제되지 않고 사용자 이벤트만 복제**
  * **Glacier/Deep Archive 객체는 복제되지 않음**
  * 삭제는 기본적으로 복제되지 않음(삭제 복제 옵션 활성화 가능)

* 사용 사례:

  * SRR: 로그 집계, PROD/TEST 동기화, 데이터 주권을 유지하면서 복원력 확보
  * CRR: 글로벌 복원력 향상, 지연시간 감소

## S3 암호화(S3 Encryption)

* “버킷”이 아니라 **버킷 내 객체(Object)**가 암호화됨

* 저장 시 암호화(At-rest) 방식:

  * **클라이언트 측 암호화**: 클라이언트가 전송 전에 암호화
  * **서버 측 암호화**: S3가 저장 시점에 암호화(전송은 평문 형태로 도착)

* 둘 다 전송 구간은 TLS 등 **In-transit 암호화** 사용 가능/권장

* 서버 측 암호화는 사실상 필수(현재 S3는 기본적으로 자동 암호화 적용)

* 서버 측 암호화 3종:

  * **SSE-C**: 고객 제공 키

    * 키 관리는 고객 책임, S3가 암복호 수행
    * Put 시 키 제공 → 해시 저장 → 암호화 후 키 폐기
    * Get 시에도 키를 다시 제공해야 함
  * **SSE-S3 (기본)**: S3 관리 키

    * AWS가 키 관리/암복호 전부 처리
    * 객체별 고유 키로 암호화 후 그 키를 마스터 키로 다시 암호화, 평문 키는 폐기
    * **AES-256**
  * **SSE-KMS**: KMS 키 사용(고객 관리 가능)

    * KMS의 CMK로 데이터 키(DEK)를 생성해 객체 암호화
    * AWS 관리 CMK 또는 고객 CMK 선택 가능
    * **역할 분리(권한 분리)**: 버킷 관리자와 KMS 키 접근자를 분리 가능

* 기본 버킷 암호화:

  * 업로드 시 헤더로 암호화 방식 지정 가능: `x-amz-server-side-encryption`

    * SSE-S3: `AES256`
    * SSE-KMS: `aws:kms`
  * 모든 S3 버킷은 기본 암호화가 적용되며, 신규 객체는 자동으로 저장 시 암호화됨
  * 기본은 SSE-S3(요청 헤더로 오버라이드 가능)

## S3 Bucket Keys

* 원래는 객체마다 고유 DEK를 만들 때 **KMS 호출이 발생** → 비용/스로틀 위험

* 리전별 KMS 호출 한도 예: PUT/sec 5,500 / 10,000 / 50,000 등(리전에 따라 상이)

* **Bucket Key**:

  * S3가 사용할 **시간 제한 버킷 키**를 KMS에서 받아와서
  * 이후 객체별 DEK 생성 작업을 **S3 내부에서 처리**(KMS 호출을 크게 줄임)
  * 결과: **KMS API 호출 감소 → 비용 절감/확장성 개선**

* 소급 적용 아님(활성화 이후 업로드 객체부터 적용)

* 유의점:

  * CloudTrail KMS 이벤트에서 객체 ARN 대신 **버킷 ARN**이 보일 수 있음
  * KMS 이벤트 로그가 줄어듦(작업이 S3로 오프로딩)
  * SRR/CRR에서도 동작, 암호화 설정 유지
  * 평문 객체를 버킷 키 사용 버킷으로 복제하면 목적지에서 암호화되어 **ETag가 바뀔 수 있음**

## S3 Presigned URL

* 내 자격증명(권한)으로 **특정 객체에 대한 임시 접근 URL**을 발급

* 만료 시간(Expiry) 포함

* URL을 가진 사용자는 인증 없이도 해당 URL로 S3 접근(발급자 권한으로 동작)

* 다운로드/업로드 모두 가능

* 앱 서버가 Presigned URL을 발급해 사용자에게 주면, **앱 서버 부하를 S3로 오프로딩** 가능

* 고려 사항:

  * 내가 접근 권한 없는 객체에 대해선 Presigned URL 생성 불가
  * 실제 접근 시점에 발급자 권한으로 평가됨(발급 후 권한이 회수되면 URL도 사실상 실패 가능)
  * 임시 자격증명(AssumeRole)로 만든 URL은 **자격증명 만료 시 URL도 무력화**
    → 보통 장기 자격(IAM User 등) 기반이 권장되는 경우가 많음(운영 보안정책에 따라 다름)

## S3 Select / Glacier Select

* 전체 객체를 다운로드하지 않고, **객체 일부만 SQL-like 쿼리로 추출**
* 대용량(최대 5TB) 객체 전체를 내려받는 시간/전송 비용을 줄임
* 지원 포맷: **CSV, JSON, Parquet**, (CSV/JSON에 대해) **BZIP2 압축** 지원

## S3 Access Points

* 하나의 버킷을 여러 팀/기능이 공유할 때 **접근 관리 단순화**
* “버킷 정책 1개” 대신 **Access Point 여러 개**를 만들고 각자 정책/네트워크 조건을 다르게 적용
* Access Point마다 고유 엔드포인트 보유
* 생성 예(CLI):
  `aws s3control create-access-point --name <name> --account-id <account-id> --bucket <bucket-name>`
* Access Point 정책만으로 끝이 아니라, 버킷 정책에도 필요한 권한이 반영돼야 함

  * 보통 버킷 정책은 넓게 허용하고, Access Point 정책에서 세분화(위임)하는 패턴

## S3 Block Public Access

* 계정/버킷/액세스포인트 단위로 퍼블릭 접근을 통제하는 설정 세트
* 옵션:

  * **IgnorePublicAcls**: 퍼블릭 ACL 생성/수정 무시(기존 ACL 자체는 남아도 효과는 제한)
  * **BlockPublicAcls**: 퍼블릭 ACL의 효과를 차단
  * **BlockPublicPolicy**: 퍼블릭 허용 버킷 정책의 생성/수정 차단(기존 정책은 유지될 수 있음)
  * **RestrictPublicBuckets**: 비 AWS 서비스 또는 비인가 사용자(일부 IAM 포함)가 퍼블릭 접근하지 못하게 제한

## S3 비용 절감 포인트

* **S3 Select/Glacier Select**: 필요한 부분만 가져와 네트워크/CPU 비용 절감
* **Lifecycle Rule**: 티어 전환/만료로 저장 비용 최적화
* **압축(Compression)**: 저장 용량 절감
* **Requester Pays**:

  * 기본은 버킷 소유자가 저장/전송/요청 비용 부담
  * Requester Pays 버킷은 **요청자**가 요청/다운로드 비용 부담
  * 소유자는 저장 비용만 부담
  * 대규모 데이터셋을 타 계정에 제공할 때 유용
  * 버킷 정책 필요
  * AssumeRole 요청이면 그 **역할의 소유 계정**이 비용을 낼 수 있음

## S3 Object Lock

* 신규 버킷에서 활성화 가능. 기존 버킷은 활성화에 AWS Support가 필요할 수 있음
* 활성화 시 **버전닝도 함께 활성화**
* Object Lock이 켜진 버킷은 **비활성화 불가**, 버전닝도 Suspend 불가(강제 유지)
* WORM(Write Once Read Many): 한번 기록된 객체 버전을 **수정/덮어쓰기 불가**
* 보존 관리 방식 2가지:

  * Retention Period
  * Legal Hold
* 객체 버전별로 둘 다 설정 가능(둘 중 하나만도 가능)

### Retention Period

* 일/년 단위로 보존 기간 설정
* 모드 2가지:

  * **Compliance 모드**:

    * 삭제/덮어쓰기/기간 단축 불가
    * 루트 사용자도 기간 단축/모드 변경 불가
    * 규정 준수 목적
  * **Governance 모드**:

    * 기본적으로 삭제/덮어쓰기 불가
    * 특정 주체에 `s3:BypassGovernanceRetention` 권한을 주면 예외적으로 변경 가능
    * 콘솔/요청에서 `x-amz-bypass-governance-retention:true`로 우회 가능(권한 필요)

### Legal Hold

* 보존 기간 없이 **On/Off**만 존재
* Legal Hold가 켜진 객체는 삭제/덮어쓰기 불가
* 변경 권한: `s3:PutObjectLegalHold`
* 실수 삭제 방지나 법적 보존 요구에 유용

## S3 Transfer Acceleration

* 전 세계에서 S3로 업로드할 때, CloudFront 엣지 네트워크를 활용해 업로드 경로를 최적화
* 사용 케이스:

  * 전 세계에서 중앙 버킷으로 업로드
  * 대용량(GB~TB) 정기 전송
  * 인터넷 업로드 대역폭을 충분히 못 쓰는 환경
* 버킷에서 기능을 활성화해야 하며, 활성화 후 속도 향상까지 **최대 20분** 걸릴 수 있음

## S3 Object Lambda

* S3의 `GET`, `LIST`, `HEAD` 응답을 **Lambda로 가공**해 반환
* 표준 S3 Access Point 기반으로 **Object Lambda Access Point**를 생성
* 클라이언트가 Object Lambda Access Point로 요청하면, S3가 Lambda를 호출해 결과를 변환 후 반환

## S3 정적 사이트 호스팅(Hosting Static Site on S3)

* S3로 정적 사이트 호스팅 가능
* 설정:

  * Static website hosting 활성화
  * Index document 지정
  * 권한 설정
* 퍼블릭으로 공개하려면:

  * Block Public Access 해제(필요한 범위만)
  * 객체 public read 허용 버킷 정책 부착
* **S3 Website Endpoint는 HTTPS를 지원하지 않음**
  → HTTPS가 필요하면 **CloudFront + S3** 조합을 사용
